# discover — 発見（SDD 段階0・Example Mapping）

新機能の最初の一歩。**いきなり要件やシナリオを書かない。** まず多観点で会話し、価値・ルール・具体例・疑問を洗い出す。出力は `specs/<feature>/discovery.md` のみ。

これは BDD の **Discovery** 工程。ここで固めた結果を、次の `specify` が EARS＋Gherkin に清書（Formulation）する。

## 最初に参照する

- `ubiquitous-language`（用語集）… 用語を揃える。**発見は新しいドメイン用語の主な発生源**なので、新語は用語集への追加を提案する。

## 価値の検査（最初に通す関門）

- **「誰の・何を・なぜ」を1文で言えること。** 言えないなら、まだ作る段階でない。問い直す。
- **最小の縦切り（MVP）に絞る。** 「無くても価値が出る」ものは入れない。残りは「対象外」か将来の別機能フォルダへ。

## 進め方（Example Mapping）

1つの機能を4種類に分解する。

- 🟡 **Story（機能）**: 対象機能を一文で。「誰が・何を・なぜ」。
- 🔵 **Rule（ルール）**: 守るべき受け入れ基準。→ 後の **EARS 要件**になる。
- 🟢 **Example（具体例）**: 各ルールを具体的な状況・値で表した例（正常／異常／境界）。→ 後の **Gherkin シナリオ**になる。
- 🔴 **Question（疑問）**: 未確定・前提・抜け漏れ・矛盾。**人間が答えるまで未解決として残す**（勝手に決めない）。

### 進行プロトコル（重要）

1. AI が下のレンズで多観点に🟡🔵🟢🔴の**草案**を作る。
2. **🔴疑問をまとめて人間に提示**し、回答を得る（選択肢があるものは AskUserQuestion で）。**AI は答えを発明しない。**
3. 回答で地図を更新。回答から新たな🔵🟢🔴が出たら追加する。
4. **🔴が残る間は Ready にしない・specify に進まない。** Ready 基準を満たすまで 2〜3 を反復。

## 多観点レンズ（抜け漏れと価値の検査）

各ルール・例を次のレンズで点検し、抜けがあれば 🔵🟢🔴 を足す：

- **プロダクト／価値**: 誰の何の課題か。価値はあるか。MVP は何か。**スコープ外**は何か。
- **開発／実現性**: 技術制約・設計への影響・既存との整合。
- **QA／テスト**: 異常系・境界・失敗モード（「どう壊れうるか」）。
- （必要に応じ UX ／ セキュリティ・データ ／ パフォーマンス）

## 大きすぎ／小さすぎの調整

- 🔵が多すぎる、または無関係な複数ゴールが混在 → **大きすぎ**。**複数の機能フォルダに分割**し、各フォルダで discover を回す。
- 🔵が1件しか出ない → **小さすぎ**（それは機能ではなく要件）。上位の機能に束ねる。
- 目安は 🔵 が数件〜十数件。

## Ready（準備OK）基準 — 満たすまで specify に進まない

- 🔴 未解決の疑問が **ゼロ**（すべて回答済み、または明示的に「対象外」と決定）。
- 🟡 価値が1文で言え、スコープの「含む／含まない」が明確。
- 🔵 が数件〜十数件で、各 🔵 に最低1つの 🟢。
- 満たしたら discovery.md 先頭の `Status:` を **Approved**（人間承認）にする。

## discovery.md の構成

```markdown
# <機能名>（Discovery）

Status: Draft        # 人間が承認したら Approved に変える（specify は Approved のみ着手）

## Story / 価値
誰の・何を・なぜ（1文）。提供価値。

## スコープ
- 含む / 含まない

## Example Map
### Rule 1: <ルール>
- 例: <具体例 正常>
- 例: <具体例 境界>
### Rule 2: ...

## 解決済みの疑問
- Q: ... → A: ...（決定の根拠）

## 対象外と決めたこと
```

## 記入例（shooting-session・抜粋）

```markdown
Status: Draft

## Story / 価値
プレイヤーが的を撃って反応速度と命中率を測り、上達を確認できる。

## スコープ
- 含む: 1セッションの的出現・ヒット/ミス判定・反応時間と命中率の集計・スコア保存
- 含まない: 銃の選択(0002)、成績推移の表示(0003)、認証(0004)

## Example Map
### Rule 1: 的の内側クリックはヒットで反応時間を記録
- 例: 的が出てから 320ms 後に中心をクリック → ヒット、反応時間 320ms
- 例(境界): 的の縁の 1px 内側をクリック → ヒット
### Rule 2: 的の外側クリックはミス
- 例: 的が無い座標(10,10)をクリック → ミス、反応時間の平均は不変
### Rule 3: 制限時間でセッション終了しスコア保存
- 例: 制限 30 秒経過 → 終了、命中率と平均反応時間を保存

## 解決済みの疑問
- Q: 的は同時に1つか複数か → A: MVP は常に1つ（複数は将来）
- Q: 反応時間の起点は → A: 的の出現時刻からクリックまで
- Q: 同じ的を二度クリックしたら → A: 2回目は無効（ヒット済みは再判定しない）

## 対象外と決めたこと
- 反動・連射などの銃挙動（0002 で扱う）
```

## 制約（ハーネス）

- discovery.md だけを作る。EARS・Gherkin・コードは書かない（それは specify 以降）。
- 🔴 疑問を勝手に決めない。人間に確認する。
- スコープ厳守。価値が薄い／不明なら MVP を問い直す。

## 完了の定義

Ready 基準を満たし、`Status: Approved`（人間承認）になった discovery.md。→ 次は `specify`。
